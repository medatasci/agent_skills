What The Source Provides

Model And Workflows

  • Unified CT and MRI segmentation foundation model based on VISTA3D lineage.
  • Predefined segment-everything modes: CT_BODY, MRI_BODY, and MRI_BRAIN.
  • Class-specific segmentation through label prompts.
  • Single image, batch, multi-GPU, brain MRI preprocessing, TensorRT, evaluation, and finetuning surfaces.

Agent Opportunity

  • Choose the right mode from a user goal and image context.
  • Resolve anatomy terms to valid label IDs.
  • Explain brain MRI preprocessing requirements.
  • Plan reproducible commands and provenance before execution.
  • Route report-guided ROI workflows to the existing Radiological Report to ROI skill.

ct-mri-segmentation-map

Segment one CT or MRI NIfTI volume using CT_BODY, MRI_BODY, or MRI_BRAIN.

How used
Agent checks model path, validates image, picks mode, constructs a MONAI bundle command, then records output segmentation and provenance.
Entrypoints
README single image inference; configs/inference.json
Runtime intent match
  1. User asks: "Create a segmentation map from this MRI."
  2. Agent resolves whether the image is CT body, MRI body, or MRI brain, asking one clarifying question if needed.
  3. Code verifies file readability, mode support, model weights, output directory, expected mask path, and write consent.
Recommendation: build the adapter before publishing.This is a high-value execution skill, but users need a stable wrapper that handles environment checks, model path discovery, mode validation, output contracts, provenance, and clear confirmation before writing segmentation files.
ct-mri-anatomy-specific-segmentation

Segment specific anatomy or classes using label prompt IDs.

How used
User asks for an anatomy, agent resolves candidate labels, asks for confirmation, runs or plans label-prompt segmentation.
Entrypoints
README label prompt; label_dict.json; model card pipeline
Runtime intent match
  1. User asks: "Create masks for the liver and spleen in this scan."
  2. Agent maps anatomy language to candidate label names and shows exact IDs before running anything.
  3. Code verifies the selected labels are supported for the requested modality and records the label prompt used.
Recommendation: pair this with label resolution first.The workflow is useful, but it should not rely on free-text anatomy terms at execution time. Publish after the wrapper can resolve synonyms, expose ambiguity, confirm label IDs, and prevent unsupported anatomy/modality combinations.
ct-mri-segmentation-label-finder

Search and explain anatomical labels, label IDs, modality coverage, and label-map provenance.

How used
Agent answers questions like "what label ID is brain stem?" or "which labels are available for MRI brain?" without running the model.
Entrypoints
metadata.json; label_mappings.json
Runtime intent match
  1. User asks: "What anatomy labels are available for brain MRI?"
  2. Agent searches labels, aliases, and modality notes, then returns candidates with source provenance.
  3. Code reads local label metadata only and produces deterministic JSON for agents and a readable table for humans.
Recommendation: make this skill first.This is the lowest-risk and highest-leverage starting point. It is read-only, fast, easy to test, and it improves every later execution workflow by giving agents a trustworthy way to resolve anatomy names before model inference.
ct-mri-batch-segmentation

Build a recursive, resumable input list and run segmentation over a folder of NIfTI files.

How used
Agent plans cohort segmentation, validates output paths, exposes skip rules, and helps resume interrupted runs.
Entrypoints
batch_inference.json; batch_inference_utils.py
Runtime intent match
  1. User asks: "Segment all NIfTI files in this folder and skip ones already done."
  2. Agent identifies folder scope, mode, overwrite policy, and whether this is a dry run or execution request.
  3. Code enumerates inputs, predicts outputs, checks existing masks, and returns a resumable run plan before execution.
Recommendation: publish after dry-run planning is reliable.Batch work can be very helpful, but it has larger side effects. Start with a deterministic planner that lists inputs, outputs, skipped files, estimated writes, and resume behavior before enabling execution.
ct-mri-cohort-segmentation

Run cohort segmentation with torchrun, multi-GPU config, and cached input discovery.

How used
Agent checks GPU count, compares workers to queued volumes, and constructs the multi-GPU command.
Entrypoints
README multi-GPU; mgpu_inference.json
Runtime intent match

LLM Semantic Processing Layer

LLM semantic processing should help users find and compose the right capability, but it should not replace deterministic execution, label-map resolution, file validation, or provenance capture.

Semantic task What the LLM can help with What deterministic code must verify Example
User intent to candidate skill Map broad requests such as "make a segmentation map" to candidate skills and ask clarifying questions about CT, MRI, brain MRI, anatomy-specific masks, or batch processing. Selected skill exists, modality is supported, required inputs are present, and side effects are clear. "Create a segmentation map from this MRI" -> ct-mri-segmentation-map with mode selection.
Anatomy language to label candidates Expand synonyms, acronyms, and report language into likely anatomy terms. Exact label IDs, label names, modality support, and source file provenance from label_dict.json or metadata.json. "brain stem" may suggest "Brain-Stem"; code resolves the actual ID and warns on ambiguity.
Workflow composition Decide whether the user needs generic segmentation, anatomy-specific segmentation, brain MRI preprocessing, batch segmentation, or the Radiological Report to ROI workflow. Commands, output paths, file existence, expected output naming, and restart/resume state. A report-guided request should route to Radiological Report to ROI, not only generic segmentation.
Source summarization and candidate discovery Read READMEs, papers, model cards, examples, configs, and scripts to identify functional blocks that may become skills. URLs, file paths, available commands, license terms, dependency names, and whether referenced files actually exist. The LLM can propose "batch segmentation"; code/review confirms batch_inference_utils.py and batch_inference.json.
Result explanation Explain outputs, limitations, warnings, and likely next steps in user language. Output segmentation exists, is non-empty, can be opened as NIfTI, and has recorded shape/affine/provenance. Summarize where the segmentation mask was written without inventing clinical interpretation.
Product implication: the codebase-to-agentic-skills workflow should use semantic search over README text, model cards, papers, scripts, configs, and examples to propose candidate skills. The workflow should then require deterministic evidence for each proposed entrypoint before adding it to the inventory.

Safety, Trust, And Review Notes

Must Say

  • Research and development only.
  • NIfTI image inputs may contain sensitive health data.
  • Network downloads, Docker, GPUs, and model execution need explicit user confirmation.
  • Model weights and data have separate license/terms from repository code.

Must Not Say

  • No diagnosis, treatment, triage, or clinical decision support claims.
  • No unsupported performance claims for a user's dataset.
  • No silent upload, sharing, or exfiltration of medical files.
  • No claim that a peer/source result is trusted merely because it is discoverable.
