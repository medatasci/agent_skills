## What Went Well

- The focused test exposed a second user-experience problem: even after the adapter policy recommendation was fixed, repo-maintenance evidence could still occupy the limited visible command slots in the candidate table.
- The final implementation addresses both the policy recommendation and the provisional CLI draft ordering.
- The regenerated NV-Generate-CTMR report is easier for a human reviewer to interpret because the task-oriented commands appear before maintenance evidence.

## What Could Be Improved

- The current `recommendation_scope` classifier is intentionally simple. It catches obvious pre-commit, lint, and formatting cases, but it does not yet score command relevance against a specific candidate's task terms.
- Future work should make adapter recommendations per-candidate rather than shared across all hypotheses when a repo contains several unrelated workflows.
- The generated wide candidate table is still dense; compact summaries help, but the detailed table could eventually be split into one candidate detail page per hypothesis.

## Next Action

- Add per-candidate command relevance scoring tied to candidate task terms and workflow-goal terms, so a repo with multiple workflows can recommend adapters from commands that actually support each specific candidate.
